= ESP32NUT
:toc:
:toclevels: 2
:sectanchors:

ESP32NUT (ESP32-NUT) is a downstream Network UPS Tools (NUT) application for
ESP32-S3 hardware. It connects a USB HID uninterruptible power supply (UPS) to
the local network and provides read-only NUT monitoring plus an authenticated
HTTPS administration console.

This is the application guide for the ESP32-NUT project. The Markdown
companion is link:README.md[README.md]. Current device facts and active work
are maintained in link:docs/ESP32_CURRENT_STATUS.md[ESP32_CURRENT_STATUS.md].

== Description

ESP32NUT uses a layered design to connect the UPS, the NUT driver, the network
server, and the administration interface:

----
USB HID UPS
    |
    v
ESP32-S3 USB host -> NUT HID driver -> read-only upsd on TCP 3493
    |
    +-> Wi-Fi -> HTTPS administration on TCP 443
----

The current application is intentionally read-only. It reports UPS state and
telemetry but does not issue power-control commands, writable NUT variables,
or shutdown commands to the UPS.

== What the application provides

* USB HID UPS discovery and polling through the ESP32-S3 native USB host
* CyberPower HID support, currently validated with a CyberPower CST150UC2
* Read-only NUT access on TCP port `3493`
* LAN-only HTTPS administration on TCP port `443`
* Per-device ADMIN authentication with secure sessions, CSRF protection, and
  failed-login throttling
* Named API tokens scoped to the Agent OTA installation route
* Wi-Fi provisioning, recovery, diagnostics, and device-owned time settings
* Authenticated local firmware checking and installation on supported builds

TCP port `8080` is a retired unauthenticated development service and must
remain closed. NUT access is read-only; UPS control operations are not part of
the current application.

== Current operating model

The published operational baseline is `v2.6.0`. This release adds an
ADMIN-session and CSRF-protected local image check before install, preserves
the authenticated install and scoped Agent OTA paths, rejects invalid images,
and retains ESP32-S3 bootloader rollback behavior. It passed target
rollback/persistence validation on `.173`. See
link:docs/ESP32_CURRENT_STATUS.md[ESP32_CURRENT_STATUS.md] for the current
branch, installed firmware, device observations, and next action.

The normal browser path uses a Synology NAS as a LAN reverse proxy:

----
Browser -- HTTPS with the trusted public certificate --> Synology :443
                                                          |
                         HTTPS; device certificate accepted by policy
                                                          v
                                                     ESP32NUT :443
----

The browser sees the Synology certificate for the configured hostname. The
ESP32 backend still uses its device-specific self-signed certificate, and the
wildcard private key remains on Synology. The complete AdGuard DNS rewrite,
Synology rule, forwarded headers, verification checklist, and DHCP recovery
procedure are in
link:docs/ESP32_MANAGEMENT_PROXY.md[ESP32_MANAGEMENT_PROXY.md].

== Supported hardware

The validated downstream target is:

* YD-ESP32-23 development board
* ESP32-S3-WROOM-1-N16R8 module
* 16 MB flash with 8 MB octal PSRAM
* Direct USB Serial/JTAG development connection
* Native USB host connection to the UPS

The inherited `boards/partitions/nut_fat_8MB.csv` partition table is intentionally retained for
the initial ESP-IDF 6.0.2 build-and-boot baseline. It uses only the lower 8 MB
of flash; expanding the layout is a separate change because dual-OTA capacity
and recovery must remain bootable.

On YD-ESP32-23 boards, verify the exact board revision before changing the
`USB-OTG` solder jumper. The native connector's data lines are wired to the
ESP32-S3, but its VBUS behavior depends on the jumper and external 5 V sources
must not be connected in a way that can backfeed either USB host connector.

== Installing

=== Software requirements

The validated local build environment is:

* ESP-IDF v6.0.2
* Python 3.x and the ESP-IDF tools
* A USB data connection to the ESP32-S3 board
* A USB HID UPS and its USB connection to the ESP32-S3 host port

For physical and network checks before working on a board, use
link:docs/ESP32_PREFLIGHT.md[ESP32_PREFLIGHT.md].

=== Build and initial flash

Use the ESP-IDF version validated by this repository:

[source,bash]
----
. /.espressif/v6.0.2/esp-idf/export.sh
idf.py set-target esp32s3
idf.py build
idf.py -p <USB-port> flash
----

Replace `<USB-port>` with the currently discovered USB device path. USB paths
and board IP addresses are session observations, not permanent coordinates.
Do not record private credentials, tokens, cookies, private keys, or
`Authorization` headers in the repository or in a support report.

For a published release, obtain the firmware and matching checksum sidecar
from the project's
link:https://github.com/BillyFKidney/esp32-nut-server/releases[GitHub
releases] and verify the checksum before installation. The published v2.6.0
artifact includes the matching `nut-esp32s3.bin.sha256` sidecar.

=== First Wi-Fi connection

Wi-Fi credentials are not compiled into the firmware. On a new device, or when
the saved credentials cannot connect, ESP32NUT starts an open setup access
point named `ESP32-NUT-<MAC suffix>` at `192.168.4.1`.

Connect to that access point and use the captive portal to select or enter the
local Wi-Fi network. The portal stages the submitted credentials and restarts
the device to test them in station-only mode. A valid DHCP address promotes
them to the saved configuration. If association or DHCP fails, the previous
active credentials remain in place, the failed pending state is cleared, and
the setup access point returns.

The setup access point is intentionally unauthenticated. Complete provisioning
from a trusted physical location and do not leave the device in setup mode
where untrusted people can join the network.

== Upgrading

Published releases should be installed from the matching release asset after
verifying its checksum. The authenticated HTTPS administration console also
supports the project's local firmware-check and installation workflow on
builds that expose it.

The administration path is protected by the per-device ADMIN session and CSRF
checks. Agent-driven installation uses a named API token limited to the
`ota.install` route. It cannot authenticate token management or unrelated
ADMIN routes. Do not use the retired TCP `8080` service for firmware updates.

The existing dual OTA slots and bootloader rollback behavior must remain
bootable after an upgrade. Treat a development image as a controlled test
installation, not as a general release.

== Configuring and using

=== HTTPS administration

The first HTTPS administration visit requires the owner to create the
per-device ADMIN password twice. Passwords are stored as salted
PBKDF2-HMAC-SHA-256 verifiers. Authenticated browser sessions use Secure,
HttpOnly, SameSite cookies and expire after fifteen minutes of idle time.

The administration console provides:

* Dashboard and device status
* Date, time, NTP, and time-zone settings
* Wi-Fi scanning, credential staging, and reconnect
* ADMIN password management
* Named API token creation and deletion
* Authenticated local firmware checking and installation

The direct device certificate is self-signed and device-specific. A direct
IP-address connection can therefore show a browser certificate warning. In
the normal local deployment, use the device hostname through the Synology
reverse proxy so the browser sees the trusted public certificate. This does
not place the wildcard private key on the ESP32.

=== Wi-Fi recovery

Holding **BOOT** for three seconds during startup clears the saved Wi-Fi
configuration and returns the device to provisioning. Continuing to the
fifteen-second factory-reset threshold clears Wi-Fi, administrator
credentials, API credentials, device identity, and the device HTTPS
certificate while preserving the firmware and OTA recovery slot.

If a factory reset regenerates the device certificate, the reverse proxy's
accepted backend certificate may need to be refreshed. Do not erase
certificate NVS blobs as a migration shortcut.

=== NUT network access

The NUT server listens on the LAN at TCP port `3493`:

[source,bash]
----
upsc cyberpower@<esp32-hostname-or-ip>
----

The current server accepts read-only `GET` and `LIST` requests. It does not
configure NUT users and does not authorize `SET`, `INSTCMD`, or `FSD`
operations. Keep port `3493` inside the trusted management network.

The general NUT identifier form is:

----
<upsname>@<hostname>[:<port>]
----

Use the current DHCP-discovered address or a local DNS name for
`<hostname>`. Do not treat historical device IP addresses as permanent.

== Security boundaries

The following boundaries are intentional:

* Management is LAN-only HTTPS on port `443`.
* Browser state-changing requests require the ADMIN session and CSRF header.
* API tokens are shown once, stored as verifiers, and scoped to Agent OTA.
* The setup access point is open only during provisioning or recovery.
* NUT access is read-only on port `3493`.
* The retired unauthenticated port `8080` remains refused.
* The shared wildcard certificate private key is never provisioned to a board.

For the detailed route, credential, and deployment rules, read
link:docs/ESP32_SECURITY.md[ESP32_SECURITY.md].

== Architecture

The main runtime layers are:

----
ESP32 application
├── Wi-Fi provisioning and station management
├── USB host event handling
│   └── USB HID device driver
├── NUT HID driver
│   └── CyberPower HID subdriver
├── Read-only NUT server
└── HTTPS administration and OTA management
----

The ESP32 USB backend is designed for the ESP32-S3 host controller. The
validated CyberPower path uses polling and reports standard UPS telemetry such
as battery charge, runtime, voltage, load, power, serial number, and `OL`
status.

== Supported UPS devices

The currently validated UPS is:

* CyberPower CST150UC2 (`0764:0601`, USB HID, polling mode)

Validation covers cold boot with the UPS present or absent, HID discovery,
regular telemetry polling, clean disconnect and reconnect, stale-data
reporting, and resumed live telemetry without restarting the NUT server.

Other USB HID UPS devices may work through inherited NUT driver support, but
they are not considered validated until tested on the ESP32-S3 target.

== Known limitations

* USB HID `SET_REPORT` is blocked; UPS commands and writable variables are not
  supported.
* Polling mode is required; interrupt-transfer and some string-read paths are
  not implemented in the ESP32 USB backend.
* The read-only NUT service does not provide TLS.
* Only USB HID devices are in scope for the current application.
* The embedded platform has limited RAM, task capacity, and POSIX support.
* DHCP is used for station-mode addressing; network addresses can change.

== Relationship to NUT

ESP32NUT is a downstream embedded application, not a general replacement for
the Network UPS Tools project. It retains NUT's driver architecture, HID
telemetry model, and network protocol where practical, while adding the
ESP32-S3 USB host, Wi-Fi lifecycle, HTTPS administration, and embedded OTA
behavior.

For upstream NUT concepts and general platform documentation, see the
link:https://networkupstools.org/[NUT project] and
link:https://networkupstools.org/documentation.html[official NUT
documentation].

== Support policy

ESP32NUT is a personal project maintained on a best-effort basis. When
reporting a problem, include the firmware version, board revision, UPS model,
the relevant status-page result, and sanitized serial or network evidence.
Never include passwords, Wi-Fi credentials, session cookies, API tokens,
private keys, or `Authorization` headers.
